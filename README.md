## Бизнес-задача

Аэропорт держится на критичной IT-инфраструктуре: киоски регистрации, система обработки багажа (BHS), табло вылета и прилёта (FIDS), посадочные гейты, СКУД, серверы. Когда что-то ломается — страдают пассажиры и задерживаются рейсы.

Проект отвечает на вопросы, которые задаёт руководство службы поддержки:

1. Как распределена нагрузка во времени и хватает ли людей в пиковые часы?
2. Какие системы создают больше всего обращений и где чинят дольше всего?
3. Насколько соблюдается SLA и где именно он срывается?
4. Как объём рейсов влияет на нагрузку службы?
5. Что снижает удовлетворённость пользователей (CSAT)?
6. Сколько рейсов задерживается из-за IT-инцидентов?

## Ключевые результаты

| Показатель | Значение |
|---|---|
| Обращений за 2 года | **33 309** (≈46 в день) |
| Решение в срок (SLA) | **95,8%** — в целом хорошо |
| Реакция в срок (SLA) | **69,1%** — главное узкое место |
| Медианное время решения | **2,8 часа** |
| Задержано рейсов из-за IT | **≈655 в год** |
| Средний CSAT | **4,4 из 5** (при нарушении SLA падает до **2,5**) |

### 1. Первичная реакция срывается в пиковые часы — это вопрос смен, а не людей «вообще»

Срывы SLA по реакции сконцентрированы в два «красных пояса»: **06:00–09:00** и **17:00–19:00** — ровно когда идут утренняя и вечерняя волны рейсов. Ночью служба справляется идеально.

![Тепловая карта срывов реакции](reports/figures/11_response_breach_heatmap.png)

### 2. Девять систем дают 80% нагрузки

Классическое Парето: рабочие станции, киоски регистрации, Wi-Fi и табло FIDS — лидеры по числу обращений. Именно там обновление парка и профилактика дадут максимальный эффект.

![Парето по системам](reports/figures/03_systems_pareto.png)

### 3. SLA по решению держится, но не для критических инцидентов

P3/P4 закрываются в срок почти всегда, а вот **P1 нарушается в 19% случаев**. Дольше всего чинят систему обработки багажа и серверы — самые «дорогие» для аэропорта отказы.

![Соблюдение SLA по приоритетам](reports/figures/05_sla_compliance_by_priority.png)

### 4. Нагрузка на поддержку растёт вместе с рейсами (r = 0,50)

Число обращений в день связано с числом рейсов — значит, нагрузку можно **прогнозировать по расписанию** и заранее планировать смены.

![Рейсы и обращения](reports/figures/08_flights_vs_tickets.png)

### 5. Нарушение SLA обваливает удовлетворённость почти вдвое

Средний CSAT при соблюдении SLA — 4,49, при нарушении — **2,51**. Скорость и есть качество сервиса.

![Драйверы CSAT](reports/figures/09_csat_drivers.png)

### 6. IT-инциденты — это ≈655 задержанных рейсов в год

Критические отказы BHS, киосков регистрации и гейтов дали 1 310 задержанных рейсов за 2 года и ≈1 160 часов простоя критичных систем.

![Влияние на рейсы](reports/figures/10_flight_impact.png)

## Рекомендации

1. **Перестроить смены 1-й линии под пиковые окна** 06:00–09:00 и 17:00–19:00 — самый быстрый способ поднять SLA по реакции.
2. **Профилактика BHS и серверов**: мониторинг и предиктивное обслуживание вместо реакции на отказ.
3. **Обновить парк киосков регистрации и рабочих станций** — лидеров по числу обращений.
4. **Прогнозировать нагрузку по расписанию рейсов** (связь подтверждена данными) и планировать ёмкость смен заранее.
5. **Контролировать переоткрытия**: проверка качества решения перед закрытием заявки — дешёвый способ поднять CSAT.

├── src/

│   ├── run_sql.py                   # прогон SQL на SQLite

## Как воспроизвести

```bash
pip install -r requirements.txt

python3 data/synthetic_data_generator.py   # 1. сгенерировать данные
python3 src/analysis.py                    # 2. построить графики и метрики
python3 src/run_sql.py                     # 3. прогнать SQL-запросы (SQLite)
python3 src/build_dashboard.py             # 4. собрать интерактивную панель
python3 src/build_notebooks.py             # 5. пересобрать ноутбуки (по желанию)
```

Зерно генератора зафиксировано (`SEED = 42`), поэтому все цифры и графики воспроизводятся один в один.

## Навыки, которые демонстрирует проект

- **Python / pandas**: обработка 33 тысяч записей, производные признаки, сводные таблицы, воспроизводимый конвейер;
- **SQL**: агрегаты, JOIN, CTE, оконные функции (накопленная доля в Парето), именованные запросы с автопрогоном;
- **Визуализация**: matplotlib/seaborn, тепловые карты, Парето, диаграммы рассеяния с трендом — и интерактивная панель на Chart.js;
- **Статистика**: корреляционный анализ, сезонность, распределения;
- **Моделирование данных**: генератор реалистичных данных с сезонностью, пиками и причинно-следственными связями;
- **Бизнес-мышление**: выводы сформулированы как решения (смены, профилактика, замена парка), а не как «интересные графики».
