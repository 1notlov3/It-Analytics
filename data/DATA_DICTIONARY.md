# Словарь данных

Описание всех таблиц и полей проекта. Данные синтетические, сгенерированы `data/synthetic_data_generator.py` (зерно зафиксировано, результаты воспроизводимы).

## `data/raw/tickets.csv` — журнал обращений (главная таблица)

Одна строка = одно обращение в службу поддержки. 33 309 строк за 2024-01-01 — 2025-12-31.

| Поле | Тип | Описание |
|---|---|---|
| `ticket_id` | строка | Уникальный номер обращения (`DME-000001`) |
| `created_at` | дата-время | Момент регистрации обращения |
| `first_response_at` | дата-время | Момент первичной реакции специалиста |
| `resolved_at` | дата-время | Момент решения; пусто у открытых и отменённых |
| `status` | строка | `Закрыт`, `В работе`, `Назначен`, `Ожидание пользователя`, `Отменён` |
| `priority` | строка | Приоритет `P1` (критический) … `P4` (низкий) |
| `category` | строка | `Оборудование`, `ПО`, `Сеть`, `Периферия`, `Доступы` |
| `system` | строка | Затронутая система аэропорта (13 систем, см. справочник) |
| `criticality` | строка | Критичность системы: `Критическая`, `Высокая`, `Средняя`, `Низкая` |
| `is_critical_for_flights` | логич. | Влияет ли отказ системы на выполнение рейсов |
| `zone` | строка | Зона аэропорта (регистрация, гейты, выдача багажа и т. д.) |
| `channel` | строка | Канал обращения: телефон, почта, портал, лично, мониторинг |
| `reporter_role` | строка | Кто обратился (авиакомпания, служба безопасности и т. д.) |
| `team` | строка | Команда-исполнитель (1-я линия, 2-я линия, полевая, вендор) |
| `agent_id` | строка | Условный сотрудник (`AG-101`…) |
| `first_response_min` | целое | Время первичной реакции, минуты |
| `resolution_min` | число | Время решения, минуты; пусто у открытых/отменённых |
| `sla_response_target_min` | целое | Норматив реакции по приоритету, минуты |
| `sla_resolution_target_min` | целое | Норматив решения по приоритету, минуты |
| `sla_response_met` | логич. | Уложилась ли реакция в норматив |
| `sla_resolution_met` | логич. | Уложилось ли решение в норматив |
| `sla_breached` | логич. | Нарушен ли SLA по решению (инверсия предыдущего) |
| `reopened` | логич. | Переоткрывалась ли заявка после закрытия |
| `service_downtime_min` | целое | Простой сервиса, минуты (только критические инциденты) |
| `delayed_flights` | целое | Число рейсов, задержанных из-за этого инцидента |
| `csat` | число 1–5 | Оценка удовлетворённости; заполнена ≈44% заявок |

## Заложенные в генератор закономерности

Чтобы анализ был содержательным, генератор моделирует реальные эффекты:

- суточный профиль с двумя волнами (утро/вечер) — как банки рейсов в аэропорту;
- недельная и годовая сезонность (летний пик, декабрь) и рост ~9% за два года;
- число обращений в день связано с числом рейсов (r ≈ 0,5);
- время реакции растёт с нагрузкой дня и «давлением очереди» в пиковые часы;
- время решения зависит от категории, критичности системы и команды (вендор — дольше всех);
- критические инциденты P1 на системах, влияющих на рейсы, дают простой и задержки рейсов;
- CSAT падает при нарушении SLA, переоткрытии и «впритык» решённых заявках;
- часть свежих заявок остаётся открытой (живой бэклог), ~1% — отменяется.
